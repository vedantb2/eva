# Sandbox Preview Proxy Setup ## Context I'm building Eva, an AI assistant platform. The app uses: - **Next.js** (App Router) - **WorkOS AuthKit** for authentication (`@workos-inc/authkit-nextjs`) - **Convex** for real-time data - **Daytona SDK** (`@daytonaio/sdk`) for remote sandboxes Users have sandboxes running on Daytona where a Next.js dev server runs on port 3000 inside the sandbox. I need to show a live preview of that dev server in my app. ## The Problem I'm currently using an iframe pointing directly at the Daytona preview URL: `tsx <iframe src="https://3000-{sandboxId}.proxy.daytona.works" /> ` This breaks because: 1. AuthKit's hosted login page (`*.authkit.app`) sets `frame-ancestors` CSP that only allows Replit, Lovable, Bolt, and v0 — not my domain 2. When any protected route in the iframe triggers an AuthKit redirect, the browser blocks it with: `Framing 'https://....authkit.app/' violates the following report-only Content Security Policy directive: "frame-ancestors ..."` 3. Cross-origin cookies and CORS issues between my app and Daytona's preview URL ## The Solution Proxy the Daytona sandbox preview through my own app so the iframe stays same-origin. Auth is handled by AuthKit middleware on my domain BEFORE the proxy forwards to Daytona. The sandbox never sees auth. `Browser → eva.com/sandbox/{id}/* → AuthKit middleware → proxy → Daytona preview URL` The iframe just points at my own orithe sgin: `tsx <iframe src="/sandbox/{sandboxId}/" /> ` ## What I Need You To Build ### 1. Custom Next.js Server (`server.ts`) Replace the default Next.js dev server with a custom Node.js HTTP server that: - Handles normal Next.js requests as usual - Intercepts `/sandbox/{sandboxId}/*` requests and proxies them to the Daytona preview URL - **Handles WebSocket upgrades** for `/sandbox/{sandboxId}/*` paths — this is critical for Next.js HMR (hot module reload) inside the sandbox. The sandbox dev server uses `/_next/webpack-hmr` over WebSocket. - Uses `http-proxy` (`npm install http-proxy @types/http-proxy`) for proxying ### 2. Daytona Preview URL Resolution (`lib/daytona-proxy.ts`) Create a module that: - Takes a `sandboxId` and port (default 3000) - Calls `sandbox.getPreviewLink(port)` via the Daytona SDK to get the preview URL and auth token - Caches the result in-memory for 60 seconds (the preview URL is dynamic and can change) - Exports a function like `getPreviewInfo(sandboxId: string, port?: number): Promise<{ url: string; token: string }>` The Daytona SDK usage looks like: `ts import { Daytona } from '@daytonaio/sdk'; const daytona = new Daytona({ apiKey: process.env.DAYTONA_API_KEY! }); const sandbox = await daytona.get(sandboxId); const preview = await sandbox.getPreviewLink(3000); // preview.url = "https://3000-{sandboxId}.proxy.daytona.works" // preview.token = "auth-token-for-private-sandboxes" ` ### 3. Proxy Logic in the Custom Server For HTTP requests to `/sandbox/{sandboxId}/*`: - Resolve the Daytona preview URL using the cache/SDK - Strip the `/sandbox/{sandboxId}` prefix from the path before forwarding - Add `Authorization: Bearer {token}` header for private sandbox access - Strip `x-frame-options` and `content-security-policy` from the proxy response headers so the iframe works - Pass through all HTTP methods (GET, POST, PUT, DELETE, etc.) For WebSocket upgrades to `/sandbox/{sandboxId}/*`: - Same URL resolution and path stripping - Use `http-proxy`'s `.ws()` method to proxy the WebSocket connection - Add the auth token header - This enables HMR to work inside the iframe ### 4. Preview Component (`components/SandboxPreview.tsx`) A React component that renders the iframe: ``tsx <iframe src={`/sandbox/${sandboxId}/`} /> `` - Full width/height, no border - Takes `sandboxId` as a prop ### 5. Update `package.json` Scripts Update the dev and start scripts to use the custom server instead of `next dev` / `next start`. ### 6. AuthKit Middleware Compatibility The existing AuthKit middleware in `middleware.ts` should continue to work. The custom server handles proxying at the Node.js HTTP level, BEFORE Next.js middleware runs for proxy routes. For non-proxy routes, Next.js middleware (including AuthKit) runs as normal. However, if AuthKit middleware currently runs on all routes, make sure `/sandbox/*` routes are still protected — the user must be authenticated before the proxy forwards the request. You may need to verify the session in the custom server's proxy handler by checking the `wos-session` cookie. ## File Structure `server.ts                          # Custom Node.js server lib/daytona-proxy.ts               # Daytona preview URL resolution + cache components/SandboxPreview.tsx       # Preview iframe component` ## Important Notes - The sandbox repo (the user's project running inside Daytona) does NOT need any changes. It just runs `next dev` normally. - Only Eva's codebase needs these changes. - The Daytona preview URL format is `https://{port}-{sandboxId}.proxy.daytona.works` - Environment variable `DAYTONA_API_KEY` must be set - Install dependencies: `npm install http-proxy @types/http-proxy` - The custom server should work for both development (`NODE_ENV=development`) and production ## Do NOT - Do not remove or break existing AuthKit setup - Do not modify the sandbox/user's project code - Do not use `next dev` directly anymore — always go through the custom server - Do not skip WebSocket proxy handling — HMR must work
